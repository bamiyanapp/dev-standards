# Code Review

目的

今回の差分のみを以下の固定観点でレビューし、問題があれば修正を促す。

## レビュー観点

- **要件との整合性**: ユーザーの要望やGoalを満たしているか
- **潜在バグ**: エラー処理漏れや境界値の不具合がないか
- **テスト不足**: 追加したロジックに対するテストコードが不足していないか
- **可読性・命名**: 命名規則に従い、他人が読みやすいコードになっているか
- **将来の変更耐性**: 既存設計を壊さず、変更が容易な構造か
- **セキュリティ懸念**: 脆弱性を生むような実装がないか
- **ドキュメント整合性**: 変更がAPI（REST/WebSocket）・DBスキーマ・主要画面に影響する場合、README/docsの更新が必要かを確認したか
- **肥大化**: 変更後、既存ファイルが著しく大きくなっていないか（過去に分割済みのファイルが再び肥大化していないか）を確認したか
- **E2Eとデプロイタイミングの整合性**: 今回の変更でバックエンドのAPIレスポンス形状・WebSocketブロードキャスト内容を変更した場合、E2Eテストが実際にデプロイ済みの（＝本PRのバックエンド変更を含まない）環境に対して実行される構成になっていないかを確認する。該当する場合、E2Eのアサーションは新しい挙動ではなく現在デプロイ済みの挙動に合わせて書き、CD実行後に値を更新する旨をコメントで明記する（またはCD完了まで`test.skip`で保留する）。デプロイ済み挙動を確認せず新しい挙動の値でアサーションすると、マージ直後のCIが必ず失敗する

指摘が1件でもあれば必ず修正し、再度Verifierプロセスにかけること。
指摘が0件になるまでレビューと修正を繰り返す。
