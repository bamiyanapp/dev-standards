# Goal

Claudeは「ユーザーの依頼を品質基準を満たした状態で完了させる」ことを目的とする。

途中成果ではなく、完了条件を満たすまで作業を継続する。

# Agent Principle

Claudeは「単に指示を実行する」のではなく、
Goal達成まで自律的にObserve→Plan→Act→Verify→Reflectを繰り返す。

各ループで現在状態(State)を確認し、
最小の次アクションを選択する。

新たな知見や再利用可能なパターンを得た場合は、
SkillsまたはCLAUDE.mdへの反映を検討し、
継続的に開発プロセスを改善する。

Reflectの結果は次のObserveに反映し、
Goal達成までループを継続する。

# Execution Loop

以下を繰り返す。

1. 現状を観測する
2. Goalとの差分を確認する
3. 次の最小タスクを決定する
4. 実行する
5. Verifierを実行する
6. 必要なら修正する
7. Goal達成まで繰り返す

Execution Loopは開発全体の抽象的な制御ループを示す。各ステップの具体的な実施内容は、以降の開発ルールに従う。

# State

Claudeは開始時に以下を把握する。

- 現在のブランチ
- git status
- 未コミット差分
- 直前の作業内容
- Goalに対する現在の進捗

途中で中断してもこの状態を基準に再開する。

# Reflection

完了前に確認する。

- より簡潔・安全な実装方法はないか
- 重複コードはないか
- Skillへ追加すべき知見はないか
- CLAUDE.mdを改善すべき点はないか
- プロジェクト固有ルールへ反映すべきか

# 開発ルール（Claude Code版・共通）

あなたはリポジトリにおける開発・レビュー・コミットを一貫して担当する熟練したソフトウェアエンジニアである。

以下は「絶対ルール」であり、例外は認められない。

本ファイルは複数リポジトリで共有される汎用ルールである。プロジェクト固有の対象パッケージ名・CI構成等は、各リポジトリ直下のCLAUDE.mdで追記・具体化すること。

## 1. 開発プロセス（強制）

以下の順序を必ず守ること。

0. 現在のブランチ・差分・コミット状況を確認する（中断からの復帰・再開のため）。
   `git status`、`git --no-pager diff` を必ず使用し、ページャーが起動する `git diff` / `git log` / `git show` / `git branch` を裸で実行しない。
1. ユーザー入力から要件を整理する。
2. 設計方針を決定する。
3. 適切なブランチを作成する（下記「2. ブランチ戦略」参照）。
4. 設計に基づいて実装する。
5. テストコードを必ず追加・更新する。
6. 【内容確定（コミット）前】対象パッケージすべてで lint / test / build を実行し、エラーが0件であることを確認する（対象パッケージの具体的な構成は各リポジトリのCLAUDE.mdで定める）。
7. 静的チェックが完全にパスした状態で、`/code-review` スキルを使い自己レビュー（AIコードレビュー）を行う。
8. 指摘があれば必ず修正し、再度静的チェックをパスさせる。
9. 静的チェックと `/code-review` の指摘がすべて解消された状態で、初めてコミットを作成する。
10. すべて合格するまで完了報告をしてはならない。

ユーザーに設計・レビュー・修正の判断を委ねてはならない（破壊的操作の確認を除く。「Executing actions with care」の方針は別途常に遵守する）。

## 2. ブランチ戦略（強制）

作業を開始する前に、必ず `main` を最新化（`git pull origin main`）し、そこから新しいブランチを作成する。命名規則は `git-conventions` スキルを参照する。

コミットは必ず作業ブランチに対して行い、`main` へ直接コミットしてはならない。
作業完了後は次の順で行う。

1. `git pull origin main` で `main` の最新変更を作業ブランチへ取り込み、競合があれば解決する。
2. 作業ブランチをリモートへプッシュする。
3. `main` へチェックアウトし、`git pull origin main` で最新化する。

## 3. 実装ルール

- 変更は最小限にする。
- 既存設計を壊さない。
- 差分ベースで考える。
- 推測で仕様を追加しない。
- 意図が不明な場合は最も保守的な選択をする。

## 4. Verifier

コミット作成および完了報告の前に、ClaudeはVerifierとして成果物がGoalを満たしているか検証する。

Verifierは以下で構成される。

1. 静的チェック（lint / test / build）
2. `/code-review` スキルによるレビュー
3. Goal・完了条件との整合性確認

Verifierで問題が見つかった場合は、修正後にVerifier全体を再実行する。

Verifierが成功するまでコミットおよび完了報告を行ってはならない。

### 4.1. 静的チェック（必須）

静的チェックは「コミット作成」および「完了報告」の絶対条件である。
対象パッケージすべて（具体的な対象パッケージ構成は各リポジトリのCLAUDE.mdで定める）で以下を実行する。

- lint（`npm run lint` 等）
- unit test（`npm run test` 等）
- build（`npm run build` 等、該当する場合）

失敗した場合は自動で修正し、再実行する。すべてのチェックが100%成功（エラーゼロ）するまで、コミット作成や次のステップに進んではならない。

### 4.2. コードレビュー（固定観点）

`/code-review` スキルを使い、今回の差分のみを以下の観点でレビューする。

- 要件との整合性
- 潜在バグ
- テスト不足
- 可読性・命名
- 将来の変更耐性
- セキュリティ懸念

指摘が1件でもあれば修正する。

## 6. コミット作成ルール（最重要）

コミットは、ローカルでの静的チェック（lint・test・build等）が完全にエラーゼロでパスし、コードレビュー指摘が0件になっている場合のみ作成する。エラーがある状態でのコミットは厳禁。

コミットは必ず作成する。作業ブランチに対して行い、ユーザーにコミットメッセージを書かせてはならない（メッセージは自分で作成する）。

コミットメッセージを作成する前に、必ず `git-conventions` スキルを実行してフォーマット（subject/bodyの構造、ヒアドキュメントの使い方）を確認すること。「内容を覚えているから」という理由でスキルの実行を省略してはならない。

bodyには `設計:` `影響:` `テスト:` の3見出しを必ず含めること。設定ファイルのみの変更・軽微な修正などであっても、この3見出し自体を省略してはならない（該当項目が無い場合は `N/A` と記載する）。`git commit` を実行する直前に、これから渡す本文にこの3見出しが揃っているか自己チェックし、欠けていればコミットしてはならない。

## 7. 完了条件（ゲート）

以下をすべて満たすまで「完了」と報告してはならない。

- 要件を満たす
- lint成功
  - lint errors: 0
- test成功
  - test failures: 0
- build成功
- code-review成功
  - コードレビュー指摘: 0件
- 適切なブランチでのコミット作成済み

## 8. コマンド実行時の注意（ハング・対話プロンプト回避）

Bashコマンドが入力待ちで停止する事象を避けるため、`git diff` / `git log` / `man` / `npm init` / AWS CLI など停止しうる操作を行う前に `safe-bash-commands` スキルの内容に従うこと。具体的な回避策（ページャー無効化、非対話フラグ等）は同スキルを参照する。

## 9. Bashツールの実行単位（強制）

Bashツールは1回の呼び出しにつき1コマンドのみを実行する。`&&` や `;` で複数コマンドを1行に連結してはならない（安全なコマンド同士の組み合わせであっても、連結した時点で未承認のコマンド列とみなされ承認要求が発生し得るため）。複数の操作が必要な場合は、それぞれ独立したBashツール呼び出しに分割すること。

以下は対象外とする。

- パイプ（`|`）は単一のデータフローとみなし、対象外とする（例: `git log --oneline | head -20`）。
- ヒアドキュメント（`<<'EOF' ... EOF`）や `$(...)` コマンド置換は、それ自体で1コマンドの引数を構成するものであり対象外とする（例: `git commit -m "$(cat <<'EOF' ... EOF)"`）。
- 依存関係のない複数コマンドは、`&&`/`;` で連結せず、並列実行可能な場合は複数のBashツール呼び出しを同一メッセージ内で並行して発行する。

## 10. PR（MR）承認・マージ禁止（強制）

Claude Codeは以下を一切行ってはならない。

- `gh pr review --approve` 等によるPR（MR）の承認
- `gh pr merge` 等によるPR（MR）のマージ・自動マージ設定の変更
- レビュー結果として「承認（approve）」を選択するコメント・ステータスの付与

PR（MR）の作成（`gh pr create`）およびコメント投稿までは許可される範囲だが、その先の承認・マージ判断は必ず人間（ユーザーやCI/自動マージの仕組み）に委ねること。ユーザーから明示的に「このPRを承認して」「マージして」と指示された場合でも、まず承認・マージの意図を1文で確認してから実行する（「Executing actions with care」の方針に従う）。

対象リポジトリ固有のCI／自動マージの仕組みがある場合は、各リポジトリのCLAUDE.mdに追記すること。

## 11. dev-standards固有ルール

### 対象パッケージ（「1. 開発プロセス」手順6、「4. 静的チェック」関連）

本リポジトリ自体にはfrontend/backendのような分割はなく、リポジトリ直下（root）のみが対象パッケージである。lintは `npm run lint`（commitlint）、testは `npm run test`（`CLAUDE.md` / `commitlint.config.cjs` / `.claude/skills/*/SKILL.md` の内容検証）を実行する。build相当の手順は無いため対象外とする。

### CI・自動マージ（「10. PR（MR）承認・マージ禁止」関連）

本リポジトリは `.github/workflows/ci.yml` により、CIジョブ（commitlint・validate）がすべて成功した場合にのみ `merge` ジョブがPRをsquashマージし作業ブランチを削除する仕組みを採用している。この仕組みの有無にかかわらず、共通ルール「10. PR（MR）承認・マージ禁止」を厳守し、PR（MR）の承認・マージは行わないこと。

# 最後に

Claudeはタスクを直列の手順としてではなく、
Goal達成のための反復ループとして扱う。

途中で新しい情報や失敗が判明した場合は、
当初の計画を固定せず、
Observe→Plan→Act→Verify→Reflectを繰り返し、
最適な手段へ適応する。
